# セキュリティ

Adobe Commerce [Pro プランのアーキテクチャ ](pro-architecture.md)は、安全性の高い環境を提供するように設計されています。 各顧客は、他の顧客とは分離された独自の隔離されたサーバー環境にデプロイされます。 本番環境のセキュリティに関する詳細は以下のとおりです。

## Web ブラウザー

クラウド環境に出入りするトラフィックの大部分は、消費者のweb ブラウザーから流入しています。 web サイトのあらゆるページで、消費者のトラフィックをHTTPSで保護できます（共有SSL証明書または追加料金で顧客自身のSSL証明書を使用）。 チェックアウトページとアカウントページは、常にHTTPSを使用して提供されます。 ベストプラクティスは、すべてのページをHTTPSで提供することです。

## コンテンツ配信ネットワーク

Fastlyは、コンテンツ配信ネットワーク（CDN）と分散型サービス拒否（DDoS）攻撃対策を提供します。 Fastly CDNは、オリジンサーバーへの直接アクセスを分離するのに役立ちます。 パブリック DNSはFastly ネットワークのみをポイントします。 FastlyのDDoS ソリューションは、非常に破壊的なレイヤ 3攻撃およびレイヤ 4攻撃、そしてより複雑なレイヤ 7攻撃から保護します。 レイヤ 7攻撃は、HTTP/HTTPS リクエスト全体に加え、ヘッダー、クッキー、リクエストパス、クライアント IPなどのクライアントやリクエストの条件、ジオロケーションなどの指標にもとづいたカスタムルールを使用してブロックすることができます。

[Fastly サービスの概要](../cdn/fastly.md)を参照してください。

## Web アプリケーションファイアウォール

追加の保護のために、FastlyのWeb アプリケーションファイアウォール（WAF）が使用されます。 FastlyのクラウドベースのWAFは、OWASP Core Rulesetなどの商用およびオープンソースのサードパーティ製ルールを使用しています。 さらに、Adobe Commerce固有のルールが使用されます。 顧客は、インジェクション攻撃や悪意のある入力、クロスサイトスクリプティング、データ流出、HTTP プロトコル違反、その他OWASPのトップ 10の脅威など、主要なアプリケーション層の攻撃から保護されます。

WAF ルールは、ソフトウェアのパッチが適用される前に、Adobe Commerceがセキュリティ問題に「仮想パッチ」を適用できるように、新しい脆弱性を検出した場合にManaged Servicesによって更新されます。 FastlyのWAFでは、レート制限またはボット検出サービスは提供されていません。 必要に応じて、顧客はFastlyに対応したサードパーティ製ボット検出サービスのライセンスを取得できます。

[Web Application Firewall （WAF） ](../cdn/fastly-waf-service.md)を参照してください。

## 仮想プライベートクラウド

Adobe Commerce Pro プランの本番環境は、バーチャルプライベートクラウド（VPC）として構成されているため、本番サーバーは隔離され、クラウド環境との間の接続が制限されています。 クラウドサーバーへは安全な接続のみが許可されます。 ファイル転送には、SFTPやrsyncなどのセキュアなプロトコルを使用できます。

顧客は、アプリケーションとの通信を保護するためにSSH トンネルを使用できます。 AWS PrivateLinkへのアクセスは、追加料金で提供できます。 サーバーへのこれらすべての接続は、環境への接続を制限するバーチャルファイアウォールである、AWSセキュリティグループを使用して制御されます。 顧客の技術スタッフは、SSHを使用してこれらのサーバーにアクセスできます。

## 暗号化

ストレージには、Amazon Elastic Block Store （EBS）が使用されます。 EBS ボリュームはすべてAES-256 アルゴリズムを使用して暗号化されているため、保存中のデータは暗号化されています。 また、CDN サーバーとオリジンサーバー間および複数のオリジンサーバー間で送信中のデータも暗号化されます。 顧客パスワードはハッシュとして保存されます。 支払いゲートウェイの資格情報などの機密情報は、SHA-256 アルゴリズムを使用して暗号化されています。

Adobe Commerceでは、列レベルまたは行レベルでの暗号化はサポートしていません。また、データが保存中ではなく、なおかつサーバー間で送信中でもない場合の暗号化もサポートしていません。 顧客は、アプリケーション内から暗号化キーを管理できます。 システムで使用されるキーは、AWS Key Management Systemに保存され、サービスの一部を提供するためにManaged Servicesで管理する必要があります。

## エンドポイントの検出と応答

[!DNL CrowdStrike Falcon]、軽量な次世代型のEndpoint Detection and Response （EDR） エージェントが、Adobe内のすべてのエンドポイント （サーバーを含む）にインストールされます。 EDR エージェントは、リアルタイムの継続的なモニタリングと収集により、Adobeのデータとシステムを保護し、脅威の迅速な特定と対応を可能にします。

## 侵入テスト

Managed Servicesでは、Adobe Commerceクラウドシステムの侵入テストを標準アプリケーションで定期的に実施しています。 お客様は、カスタマイズしたアプリケーションの侵入テストを行う責任があります。

## 支払いゲートウェイ

Adobe Commerceでは、クレジットカードのデータが消費者のブラウザから直接支払いゲートウェイに渡されるようにするために、支払いゲートウェイの統合が必要です。 カードデータは、Adobe Commerce Pro プランのManaged Services環境では利用できません。 e コマースアプリケーションで発生したトランザクションにおけるアクションは、ゲートウェイからトランザクションを参照することで完了します。

## Adobe Commerceアプリ

Adobeでは、コアアプリケーションのコードにセキュリティ上の脆弱性がないか定期的にテストしています。 不具合やセキュリティ問題に対するパッチは顧客に提供されます。 製品セキュリティチームは、OWASPのアプリケーションセキュリティのガイドラインに従ってAdobe Commerce製品を検証しています。 コンプライアンスのテストと検証には、いくつかのセキュリティ脆弱性評価ツールと外部ベンダーを使用します。 セキュリティツールには、次のものが含まれます。

- Veracode静的スキャンと動的スキャン
- RIPS ソースコードのスキャン
- TrustwaveおよびAlert Logicの脆弱性スキャンサービス
- Burp Suite Pro
- OWASPZAP
- andSqlMap

これらのツールを使って、全コードベースを隔週でスキャンしています。 セキュリティパッチは、ダイレクトメール、アプリケーション内の通知、[ セキュリティセンター](https://helpx.adobe.com/security.html)で顧客に通知されます。

顧客は、PCI ガイドラインに従って、これらのパッチがリリースから30日以内に、カスタマイズしたアプリケーションに適用されるようにしなければなりません。 Adobeでは、定期的にサイトを監視し、既知のセキュリティリスク、マルウェア、不正アクセスに関する最新情報を受け取ることができる[ セキュリティスキャンツール ](https://experienceleague.adobe.com/en/docs/commerce-admin/systems/security/security-scan)も提供されています。 セキュリティスキャンツールは無料のサービスで、どのバージョンのAdobe Commerceでも実行できます。

セキュリティ研究者に脆弱性の特定と報告を促すために、Adobe Commerceには内部テストに加えて[ バグ報奨金プログラム ](https://hackerone.com/magento)があります。 さらに、お客様には、必要に応じて、アプリケーションの完全なソースコードが提供され、独自のレビューが行われます。

## 読み取り専用のファイルシステム

あらゆる実行コードは読み取り専用のファイルシステムイメージにデプロイされ、攻撃に使用できるサーフェスを大幅に削減します。 デプロイメントプロセスでは、Squash-FS イメージを作成し、PHPやJavaScriptのコードをシステムに注入したり、Adobe Commerceのアプリケーションファイルを変更したりする機会を減らします。

## リモート展開

実行コードをManaged Servicesの本番環境に導入する唯一の方法は、プロビジョニングプロセスで実行することです。 プロビジョニングでは、ソースコードをソースリポジトリからリモートリポジトリにプッシュすることで、デプロイメントプロセスが開始されます。 そして、デプロイメントターゲットへのアクセスは制御されているので、デプロイメントターゲットにアクセスできる担当者を完全に管理することができます。 実稼動以外の環境へのアプリケーションコードのデプロイメントは、すべて顧客が管理します。

## ログ

すべてのAWS アクティビティは、AWS CloudTrailに記録されます。 オペレーティングシステム、アプリケーションサーバー、データベースのログは、本番サーバーに保存され、バックアップサバーに保存されます。 ソースコードの変更はすべてGit リポジトリに記録されます。 デプロイメントの履歴は、Adobe Commerce [Project Web Interface](../project/overview.md)で確認できます。 サポートアクセスはすべてログに記録され、サポートセッションも記録されます。

[ ログの表示と管理](../test/log-locations.md)を参照してください。

## 機密データ

機密データには、消費者の個人情報やManaged Servicesのお客様の機密データが含まれます。 顧客および消費者の機密データの保護は、Adobe Commerce Managed Servicesの重要な義務です。 Managed ServicesとAdobeのお客様には、個人を特定できる情報に関する法的義務があります。 アーキテクチャのセキュリティ機能に加えて、機密データの配布とアクセスを制限するための他の管理機能もあります。

顧客は自身のデータを所有し、そのデータの保存場所をコントロールできます。 お客様は、実稼動インスタンスと開発インスタンスが格納される場所を指定します。 また、Commerceを使用するAdobe Commerce Reporting環境に使用する場所と、そのAdobe Commerce Reporting アプリケーションが個人を特定できる情報にアクセスできるかどうかを指定します。 実稼動インスタンスは、ほとんどのAWS リージョンに存在します。一方、開発環境とAdobe Commerce レポート環境は、現在、米国または欧州連合に存在します。

機密データはFastly CDN サーバーネットワークを通過する可能性がありますが、Fastly ネットワークには保存されません。 Managed Services製品に含まれるすべてのパートナーは、機密データの保護を保証する契約上の義務を負っています。 Managed Servicesは、お客様が指定した場所から機密性の高いお客様データまたは消費者データを移動することはありません。

Managed Services製品に含まれるサービスの提供の一環として、Managed Servicesのスタッフは、機密データを含む実稼動システムへのアクセスを必要とします。 これらの従業員は、顧客と消費者の重要なデータを保護する義務についてトレーニングを受けています。 これらのシステムへのアクセスは、アクセスが必要な従業員に限定されます。 これらの従業員は、これらのサービスを提供するために必要な時間のみアクセスできます。

## EU一般データ保護規則

一般データ保護規則（GDPR）とは、欧州連合（EU）在住の個人の個人情報の収集と処理に関するガイドラインを設定する法的枠組みです。 これらの規制は、サイトの拠点に関係なく適用され、EUの訪問者はそれにアクセスできます。

訪問者には、サイトが収集したデータが通知され、情報収集に明示的に同意する必要があります。 サイトに保存されている個人データが侵害された場合、サイトは訪問者に通知する必要があります。

サイトを運営する販売者または企業には、サイトのデータセキュリティを監督する専任のデータ保護担当者が必要です。 訪問者からデータの消去を要求された場合は、データプライバシー責任者（またはweb サイト管理チーム）に連絡する必要があります。

GDPRでは、収集した個人を特定できる情報（名前、人種、生年月日など）は、匿名化または仮名化することを求めています。

>[!NOTE]
>
>このページでは、GDPRに対して考慮すべき一般的な概要を説明します。 Adobe Commerceでの個人情報の保存方法について詳しくは、_[セキュリティおよびコンプライアンス ガイド ](https://experienceleague.adobe.com/en/docs/commerce-operations/security-and-compliance/overview)_&#x200B;を参照してください。 ビジネスが法的義務に準拠する方法を決定するには、弁護士に相談するか、[公式テキスト ](https://eur-lex.europa.eu/eli/reg/2016/679/oj)を参照してください。

## バックアップ

バックアップは、最後の24時間の操作のために1時間ごとに実行されます。 24時間が経過すると、AWS EBS Snapshot サービスを使用して、バックアップがスケジュールに従って保持されます。 [保持ポリシー](pro-architecture.md#retention-policy)を参照してください。

サービスは、冗長ストレージに対して独立したバックアップを作成します。 EBS ボリュームは暗号化されているので、バックアップも暗号化されています。 また、Managed Servicesでは、お客様の要求に応じてオンデマンドバックアップを実行します。

Managed Servicesのバックアップと復元のアプローチでは、高可用性アーキテクチャとシステム全体のバックアップを組み合わせて使用します。 各プロジェクトは、3つの個別のAWS可用性ゾーンに複製されます。各ゾーンには個別のデータセンターがあります。

[ スナップショットとバックアップ管理](../storage/snapshots.md)を参照してください。
