# ステージングと実稼動のテスト

コード、ファイル、データをステージングまたは実稼動に移行した後、環境URLを使用してサイトとストアをテストします。 次に、ログの検証、Fastly設定のテスト、ユーザー受け入れテスト（UAT）などの情報を示します。

{{second-staging}}

## ログファイル

テスト時にデプロイメントのエラーやその他の問題が発生した場合は、ログファイルを確認してください。 ログファイルは`var/log` ディレクトリにあります。

デプロイメント ログは`/var/log/platform/<prodject-ID>/deploy.log`にあります。 `<project-ID>`の値は、プロジェクト IDと、環境がステージング環境と実稼動環境のどちらに依存しているかによって異なります。 例えば、プロジェクト IDが`yw1unoukjcawe`の場合、ステージング ユーザーは`yw1unoukjcawe_stg`、実稼動ユーザーは`yw1unoukjcawe`です。

実稼動環境またはステージング環境のログにアクセスする場合は、SSHを使用して3つのノードのそれぞれにログインし、ログを探します。 または、[New Relic ログ管理](../monitor/log-management.md)を使用して、すべてのノードから集約されたログデータを表示および照会できます。 [ ログの表示](log-locations.md#application-logs)を参照してください。

## コードベースを確認する

コードベースがステージング環境と実稼動環境に正しくデプロイされていることを確認します。 環境には同じコードベースが必要です。

## 構成設定の確認

ベース URL、ベース管理者URL、マルチサイト設定など、管理パネルで設定設定を確認します。 追加の変更が必要な場合は、ローカル Git ブランチで編集を完了し、統合、ステージング、実稼動の`master` ブランチにプッシュします。

## Fastlyのキャッシュを確認

[Fastly](../cdn/fastly-configuration.md)の設定では、適切なFastly Service IDおよびFastly API トークン資格情報の使用、Fastly VCL コードのアップロード、DNS設定の更新、環境へのSSL/TLS証明書の適用など、詳細に注意する必要があります。 これらの設定タスクを完了したら、ステージング環境と実稼動環境でFastlyのキャッシュを確認できます。

**Fastly サービス設定を確認するには**:

1. ステージングおよび実稼動用の管理者に、`/admin`のURLまたは[更新された管理者URL](../environment/variables-admin.md#admin-url)を使用してログインします。

1. **Stores** > **Settings** > **Configuration** > **Advanced** > **System**&#x200B;に移動します。 スクロールして、**フルページキャッシュ**&#x200B;をクリックします。

1. **Caching application**&#x200B;の値が&#x200B;_Fastly CDN_&#x200B;に設定されていることを確認します。

1. Fastlyの認証情報をテストする。

   - 「**Fastly Configuration**」をクリックします。

   - Fastly サービス IDおよびFastly API トークン資格情報の値を確認します。 [Fastly資格情報の取得](/help/cloud-guide/cdn/fastly-configuration.md#get-fastly-credentials)を参照してください。

   - 「**資格情報をテスト**」をクリックします。

   >[!WARNING]
   >
   >ステージング環境と実稼動環境で、正しいFastly サービス IDとAPI トークンを入力したことを確認してください。 Fastlyの認証情報は、サービス環境ごとに作成され、マッピングされます。 実稼動環境でステージング資格情報を入力すると、VCL スニペットをアップロードできず、キャッシュが正しく機能せず、キャッシュ設定が間違ったサーバーとストアをポイントします。

**Fastlyのキャッシュ動作を確認するには**:

1. `dig` コマンドラインユーティリティを使用してヘッダーを確認し、サイト設定に関する情報を取得します。

   `dig` コマンドを使用すると、任意のURLを使用できます。 次の例では、Pro URLを使用しています。

   - ステージング：`dig https://mcstaging.<your-domain>.com`
   - 実稼動：`dig https://mcprod.<your-domain>.com`

   追加の`dig` テストについては、「[DNS](https://docs.fastly.com/en/guides/working-with-domains)を変更する前にテストする」を参照してください。

1. `cURL`を使用して、応答ヘッダー情報を検証します。

   ```bash
   curl https://mcstaging.<your-domain>.com -H "host: mcstaging.<your-domain.com>" -k -vo /dev/null -H Fastly-Debug:1
   ```

   ヘッダーの検証について詳しくは、[応答ヘッダーの確認](../cdn/fastly-troubleshooting.md#check-cache-hit-and-miss-response-headers)を参照してください。

1. ライブが終了したら、`cURL`を使用してライブサイトを確認します。

   ```bash
   curl https://<your-domain> -k -vo /dev/null -H Fastly-Debug:1
   ```

## UAT テストの完了

ステージングと実稼動に関するユーザー受け入れテスト（UAT）を完了します。 以下のテストは、マーチャントおよび顧客としてテストする可能性のあるタスクと領域の簡単なリストです。 カスタムモジュール、拡張機能、サードパーティ製品との統合に関する追加のテストを含め、より長いリストを作成する必要があります。 テストは、デスクトップ PC、ノートパソコン、モバイルデバイスを使用します。

問題が発生した場合は、再生手順、エラーメッセージ、奇妙なスクリーンキャプチャ、リンクを保存します。 この情報を使用して、統合環境コードおよび構成または環境設定の問題を調査および修正します。

## 負荷と負荷のテスト

開始する前に、ステージング環境と実稼動環境で大規模なトラフィックおよびパフォーマンステストを実行することをお勧めします。 フロントエンドとバックエンドのプロセスのパフォーマンステストを検討する。

テストを開始する前に、テスト対象の環境、使用しているツール、時間枠をサポートするチケットを入力します。 結果と情報でチケットを更新し、パフォーマンスを追跡します。 テストが完了したら、更新された結果を追加し、チケットテストのメモに日付とタイムスタンプが記載されます。

ローンチ前の準備プロセスの一環として、[Performance Toolkit](https://github.com/magento/magento2/tree/2.4/setup/performance-toolkit)のオプションを確認してください。

最適な結果を得るには、次のツールを使用します。

- [ アプリケーション パフォーマンス テスト ](../environment/variables-post-deploy.md#ttfb_tested_pages) - サイト応答時間をテストするために`TTFB_TESTED_PAGES`環境変数を設定して、アプリケーション パフォーマンスをテストします。
- [包囲攻撃](https://www.joedog.org/siege-home/) - ストアを制限にプッシュするためのトラフィック形成およびテスト ソフトウェア。 設定可能な数のシミュレートされたクライアントで、サイトにアクセスします。 Siegeは、基本認証、Cookie、HTTP、HTTPS、およびFTP プロトコルをサポートしています。
- [Jmeter](https://jmeter.apache.org) - フラッシュ セールスのようなスパイク トラフィックのパフォーマンスを測定するのに役立つ優れた負荷テスト。 サイトに対して実行するカスタムテストを作成します。
- [New Relic](../monitor/new-relic-service.md) （提供） – データ、クエリ、Redisなどの送信アクションごとに費やされた時間を追跡して、パフォーマンスの低下を引き起こすサイトのプロセスと領域を特定します。
- [WebPageTest](https://www.webpagetest.org)および[Pingdom](https://www.pingdom.com)：サイト ページのリアルタイム分析は、異なるオリジンの場所で読み込み時間を短縮します。 Pingdomは料金が必要な場合があります。 WebPageTestは無料のツールです。

## 機能テスト

Magento Functional Testing Framework （MFTF）を使用して、Cloud Docker環境からAdobe Commerceの機能テストを実行できます。 _Cloud Docker for Commerce ガイド_&#x200B;の[Application testing](https://developer.adobe.com/commerce/cloud-tools/docker/test/application-testing)を参照してください。

## セキュリティスキャンツールの設定

無料のセキュリティスキャンツールがあります。 サイトを追加してツールを実行するには、[ セキュリティスキャンツール ](../launch/overview.md#set-up-the-security-scan-tool)を参照してください。
